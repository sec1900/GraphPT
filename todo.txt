E3. Solo pool 下多工具被迫串行 → 调度器失去意义          ✅ 已修
    新增直接调度引擎 run_full_scan / run_scan_layer
    ThreadPoolExecutor 直接跑工具，绕过 Celery 串行限制
    Windows/Linux 行为完全一致
    现象:
      - start_services.py 启动后 worker 打印 ready
      - Redis 队列堆积任务（queue_depth=21+），永远不降
      - celery inspect ping 返回空，health API 报 worker 不在线
      - 前端调度器返回 "dispatched" 但实际什么都没发生 —— 静默失败
    触发条件: 前一次 worker 非正常退出（Ctrl+C / kill），残留 Redis 连接
    不修的后果:
      - 用户看到 Web UI 显示 dispatched、进度条在走，实际零产出
      - 只能靠人工发现（queue_depth 不降），没有任何告警
    修法:
      a) 短期: start_services.py 启动前自动执行 redis FLUSHALL（或至少清理 scheduler:* key）
      b) 中期: web health API 增加队列深度趋势检测——连续 3 次采样 queue_depth 不变 → 报警
      c) 长期: 放弃 solo pool，改为 Windows 上多进程 prefork 或 gevent（需验证可用性）
    为什么这样修:
      - a) 简单粗暴但保证每次启动干净
      - b) 把静默失败变成可见告警
      - c) solo pool 是 Windows 限制，但 gevent/thread 池可能可用
    修了的好处:
      - 消除最常见的"看起来在跑实际没跑"的假像

E3. Solo pool 下多工具被迫串行 → 调度器失去意义          ✅ 已修 (随 E2-c)
    start.bat + start_services.py 改用 --pool=threads --concurrency=10
    文件: start_services.py (--pool=solo --concurrency=4)
    现象: --pool=solo 强制 1 worker，--concurrency=4 完全无效。
         25 个工具、同层 5 个并行派发，实际只能一个一个跑。
         gobuster:dns DNS 爆破（可能跑几十分钟）阻塞整个链。
    不修的后果:
      - 调度器的"同层并行"设计被彻底绕过
      - 一次扫描可能跑数小时到数天
      - 前端看上去 5 个 dispatched，实际第 2 个在排队等第 1 个
    修法:
      a) Windows: 尝试 --pool=threads（Celery 5.x 实验性支持），或 --pool=gevent
      b) 如果线程池可用: --concurrency=10 真正并行
      c) 如果都不行: 文档化这是 Windows 限制，推荐 Docker/Linux 部署
    为什么这样修:
      - 并行是调度器设计的核心假设，串行违背设计意图
      - threads 池在 I/O 密集场景（等子进程 stdout）表现接近 prefork
    修了的好处:
      - 同层 5 工具真正并行，扫描时间缩短 5x
      - 调度器"同层并行跨层串行"的设计生效

E4. 调度器锁残留 — 崩溃后无法重新派发                    ✅ 已修
    心跳机制替代绝对超时：任务每30s更新心跳，心跳停5min自动释放
    24h长任务不受影响（只要心跳持续更新）
    新增 POST /api/scheduler/unlock 手动解锁端点
    advance_once 返回区分 "locked" vs "running"
    文件: scheduler.py (advance_once, Redis SET NX 锁 TTL 1800s)
    现象: worker 崩溃/被杀后，scheduler:lock:* 残留 30 分钟。
         重新调用 advance_once → status:"running", dispatched:[] → 不派发也不报错
    不修的后果:
      - 用户杀掉卡死的扫描想重来 → 调度器静默拒绝 30 分钟
      - 前端无任何提示，用户以为系统坏了
    修法:
      a) web API 增加 POST /api/scheduler/unlock — 用户手动解锁
      b) advance_once 返回时区分 "locked" vs "running"（当前一律返回 "running"）
      c) 锁 TTL 从 1800s 降到 300s（5分钟），加上 task 完成后主动释放（已有 _release_lock）
    为什么这样修:
      - a) 给用户一个显式的"重置"按钮
      - b) 让前端能提示"工具 xxx 正在运行中，请等待"而非沉默
      - c) 正常任务 2-3 分钟完成，5 分钟 TTL 足够，崩溃后等待成本从 30 分钟降到 5 分钟
    修了的好处:
      - 用户不再被锁卡住 30 分钟
      - 前端能区分"正在跑"和"卡死了"
